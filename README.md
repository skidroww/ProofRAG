RAG(검색 증강 생성) 시스템의 데이터 무결성, 검색 효율성, 그리고 구조 개선


1. 머클 트리(Merkle Tree)를 응용한 계층적 청크 관리 및 검색 효율화 -> 선형 체인 구조로 전환

RAG 시스템은 방대한 문서를 여러 개의 '청크(Chunk)'로 쪼개어 벡터 데이터베이스에 저장-> 이때 머클 트리의 구조를 차용하여 문서를 계층적으로 임베딩, 상위 노드(루트)에는 문서 전체의 핵심 요약 임베딩을 저장하고, 하위 노드(나뭇가지)로 갈수록 세부 단락의 임베딩을 저장하는 RAPTOR(Recursive Abstractive Processing for Tree-Organized Retrieval)와 같은 트리 기반 검색 구조를 구축 -> 특정 문서가 업데이트되었을 때 전체를 다시 인덱싱할 필요 없이 해당 하위 노드의 가지(Branch)만 해시를 갱신하여 시스템 부하를 줄인다.

-> 구현 어려워서 포기...(관계형 구조의 복잡성과 라우팅 난이도의 어려움)

-> 선형 체인 구조로 전환
머클 트리의 복잡성을 해결하기 위해 비트코인의 핵심인 선형적 해시 체인(Linear Hash Chain)을 청크 관리에 적용하는 아키텍쳐로 바꿈
-> 문서를 청크 단위로 쪼갤 때 이전 청크의 해시를 메타데이터에 포함시켜 단일 체인으로
-> 이유: 포괄적인 문맥 요약보다 구체적인 팩트, 법 위반 사실, 금액 등 조작되지 않은 '원본 데이터'의 정확성을 중요시, 선형 체인 구조는 무거운 LLM 요약 작업(비용) 없이도 데이터베이스 내 청크의 순서와 무결성을 보장한다
-> 검색된 정보가 오염 없이 도달했음을 100프로 증명할수있어 환각 원천 차단 가능


2. 타임스탬프 서버를 활용한 데이터 출처 증명 및 오염 방지 (Data Provenance)

RAG 시스템의 취약점인 출처가 불분명하거나 조작된 데이터(Data Poisoning)가 검색되어 AI가 할루시네이션(환각)을 일으키는 것 -> 문서가 데이터베이스에 추가될 때 타임스탬프와 해시 기반 체인 기록을 남기면, LLM이 문서를 참조할 때 해당 데이터의 원본 여부, 수정 이력, 시의성을 완벽하게 검증할 수 있다-> 이를 통해 항상 신뢰할 수 있는 최신 원본 문서에 높은 가중치를 부여하는 검색이 가능

3. 가장 긴 체인(합의 알고리즘) 원리를 통한 상충되는 정보의 가중치 결정

RAG 모델이 여러 문서를 검색했을 때 A 문서와 B 문서의 내용이 서로 모순되는 경우가 있다면 ->  비트코인의 다수결/작업증명에 따른 '가장 긴 체인 선택' 원리를 차용하여, 더 많은 신뢰도 높은 문서들(또는 사용자 피드백, 인용 횟수, 크로스 체크된 출처)과 일치하는 내용에 '합의 가중치(Consensus Weight)'를 부여-> 이를 통해 모순된 정보 중 시스템이 어떤 정보를 채택하여 답변을 생성할지 결정하는 로직을 강화

4. 간소화된 지불 검증(SPV)을 활용한 경량화 엣지(Edge) RAG 모델

SPV 원리처럼, 클라이언트(엣지 디바이스)는 무거운 검색과 임베딩 처리를 중앙 서버에 맡기되, 반환된 검색 컨텍스트가 신뢰할 수 있는 코퍼스(Corpus)에서 온 것인지 빠르게 확인하기 위해 가벼운 메타데이터(헤더와 해시 증명)만으로 데이터의 유효성을 검증하는 '분산형/경량화 RAG 검증 아키텍처'를 설계
